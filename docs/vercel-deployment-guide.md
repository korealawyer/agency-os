# Vercel 배포 및 Prisma 연결 트러블슈팅 가이드

본 문서는 로컬 환경에서 정상 작동하던 프로젝트를 Vercel 및 서버리스 환경으로 배포하는 과정에서 발생한 주요 에러 원인과 해결 방법을 정리한 가이드입니다. 향후 유사한 환경 구축 및 유지보수 시 참고용으로 작성되었습니다.

---

## 1. NextAuth 기반 인증 에러 (Vercel Edge 호환성)

### 🚨 발생 원인

1. **Edge 런타임 제한**
   - Vercel의 미들웨어(Middleware)는 빠른 실행을 위해 가벼운 Edge 런타임에서 동작합니다. 
   - 하지만 인증 설정 파일(`auth.ts`) 내부에 Edge 환경에서 지원하지 않는 Node.js 전용 모듈(예: `PrismaClient` 등 무거운 DB 모듈)이 포함되어 있어 미들웨어 실행 중 Crash가 발생했습니다.

2. **보안 쿠키 인식 실패 (HTTPS 환경)**
   - Vercel에 배포되면 기본적으로 HTTPS가 적용됩니다. 
   - HTTPS 환경에서 NextAuth는 보안을 위해 세션 쿠키 이름을 `next-auth.session-token`에서 `__Secure-next-auth.session-token`으로 자동 변경합니다.
   - 하지만 기존 로컬 환경 기준 코드는 예전 쿠키 이름만 찾도록 작성(`getToken()` 사용)되어 있어, 인증 정보를 제대로 읽어오지 못하는 문제가 발생했습니다.

3. **Untrusted Host (신뢰할 수 없는 호스트) 차단**
   - Vercel 배포 시 `*.vercel.app` 과 같이 동적으로 생성되는 URL에서는 NextAuth가 인가되지 않은 도메인의 요청으로 간주하여 요청을 차단하는 보호 로직이 작동했습니다.

### 🛠 해결 방법

1. **Edge 호환 `auth.config.ts` 파일 분리**
   - Edge 환경에서 작동할 수 있는 가벼운 설정(Provider 등)만 정의한 `auth.config.ts` 파일을 따로 분리했습니다.
   - 미들웨어는 무거운 DB 어댑터를 제외하고 오직 이 가벼운 파일만 참고하도록 구조를 재설계했습니다.

2. **공식 서버 함수 `auth()` 사용**
   - 쿠키 이름을 직접 파싱하다가 생기는 문제를 방지하기 위해 `getToken()` 대신 NextAuth에서 제공하는 통합 서버 인증 함수인 `auth()`를 API와 서버 컴포넌트에 전면 적용했습니다.

3. **`trustHost: true` 및 시크릿 명시**
   - Vercel 환경에서 도메인이 변경되더라도 인증 요청을 안전하게 처리할 수 있도록 NextAuth 설정 옵션에 `trustHost: true`를 강제 적용하고, `AUTH_SECRET` 환경변수를 명확히 매핑하여 문제를 해결했습니다.

## 2. Prisma 데이터베이스 연결 및 런타임 에러

### 🚨 발생 원인

1. **SSL 보안 접속 설정 누락 (Neon / 서버리스 DB 호환)**
   - 개발 단계의 로컬 환경과 다르게, Vercel 같은 클라우드 환경에서 원격 DB(Neon 등)로 접속할 때는 데이터 탈취 방지를 위해 SSL 인증(보안 터널)이 필수적입니다.
   - 해당 옵션이 DB 연결 객체 설정에서 누락되어 외부 접근이 차단되었습니다.

2. **스키마 변경 미반영(없는 DB 컬럼 접근 로직)**
   - 로그인 성공 시 사용자의 '마지막 로그인 시간(`lastLoginAt`)'을 업데이트 처리하는 로직이 있었으나, 실제 Prisma 스키마와 클라우드 DB에는 해당 컬럼이 정의되지 않아 데이터베이스 오류를 발생시키고 전체 프로세스가 중단되었습니다.

### 🛠 해결 방법

1. **커넥션 풀에 SSL 옵션 명시**
   - `@neondatabase/serverless` 의 `Pool` 객체 혹은 Prisma DB 커넥션 설정 코드에서 `ssl: { rejectUnauthorized: false }` 옵션을 추가 명시하여, 원격 서버와 서버리스 환경에서도 안전한 SSL 통신이 가능하도록 수정했습니다.

2. **비핵심/오류 유발 로직 과감한 제거**
   - 런타임 셧다운의 직접적인 원인이 된 존재하지 않는 컬럼(`lastLoginAt`)의 업데이트 코드를 즉각 삭제하여 로그인 프로세스가 정상적으로 종료될 수 있도록 견고하게 수정했습니다.
   - 기능 자체를 제거하는 것이 목적이 아니라 "안정적인 배포 상태" 확보를 최우선으로 하여 조치했습니다.

## 💡차후 개발 시 유의사항(Best Practices)

1. **배포 환경과 로컬 환경의 분리 인식**
   - 로컬에서는 `.env` 에 의존하지만, 프로덕션(Vercel)에서는 항상 Vercel Environment Variables 설정 콘솔에 빠짐없이 변수를 추가해야 합니다. 
   - API 키, 시크릿(ex: `AUTH_SECRET`), DB URL 등 검증을 철저히 진행하세요.

2. **Prisma Schema 와 실제 DB 상태 동기화**
   - `prisma/schema.prisma` 변경 시에는 항상 `npx prisma db push` 또는 `npx prisma migrate`를 실행해 실제 DB 구조를 업데이트해야 합니다.
   - 로직(코드) 상에 정의된 모델과 실제 테이블 구조의 일치 여부를 배포 전에 확인하는 체크리스트를 만드세요.

3. **NextAuth 미들웨어 구조 표준화**
   - 향후 새로운 모듈(어댑터)를 추가할 때, Edge 런타임에 호환되지 않는 패키지가 있다면 반드시 `auth.ts`가 아닌 다른 분리된 곳에 위치하여 미들웨어(`middleware.ts`)가 터지지 않게 관리해야 합니다.

## 3. Vercel 배포 플랜(Hobby) 제약에 따른 빌드/배포 에러

### 🚨 발생 원인
1. **Cron Job 지원 제한 (Hobby 플랜)**
   - 프로젝트 초기에 스케줄링(예: 매일 자정 입찰 최적화 로직 등)을 위해 `vercel.json` 파일에 `crons` 설정을 추가(`/api/cron/bid-optimization` 호출)했습니다.
   - 하지만 Vercel 프로덕션 배포 단계에서 사용 중인 계정이 **무료(Hobby) 플랜**인 경우, 해당 Cron Job 기능이 지원 범위를 초과하거나 제한되어 **배포 자체(Build)가 실패하거나 경고 에러**를 발생시켰습니다.

### 🛠 해결 방법
1. **`vercel.json` 내 Cron Job 설정 제거**
   - Hobby 플랜과 호환될 수 있도록 배포 설정 파일(`vercel.json`)에서 문제를 일으키는 `"crons": [...]` 블록을 임시로 완전히 삭제했습니다.
   - 이를 통해 애플리케이션 프론트엔드 및 기본 API 배포가 우선적으로 정상 처리되도록 조치했습니다.
   
2. **차후 대안 준비 (Best Practice)**
   - Vercel 환경에서 Cron 기능이 꼭 필요하다면 **Pro 플랜 이상의 업그레이드**를 고려해야 합니다.
   - 또는 GitHub Actions의 Schedule(Cron) 트리거, 외부 크론 스케줄링 서비스(예: cron-job.org, Upstash Redis 등)를 연동하여 Vercel API 엔드포인트를 주기적으로 호출하는 **외부 우회 방식(Webhook)**을 설계하는 것이 좋습니다.
